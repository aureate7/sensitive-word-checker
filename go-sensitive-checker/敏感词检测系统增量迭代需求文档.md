# 敏感词检测系统增量迭代需求文档

## 1. 文档目的

本文档用于描述现有“敏感词检测系统”的后续版本迭代需求。
当前系统主体功能已经完成，能够实现基础敏感词检测、词库管理及接口服务。后续开发目标不是从零重构系统，而是在 **保留现有工程主体不变** 的前提下，补充更具学术性和创新性的检测能力，用于提升项目在原创开源软件赛道中的竞争力。

本文档主要用于指导后续代码修改、模块扩展、功能增强和比赛材料包装。

### 2.1 已完成内容

当前系统已经具备以下能力：

1. 基础敏感词检测功能
2. 敏感词词库管理功能
3. 对外 API 服务接口
4. 基于现有算法的数据匹配能力
5. 基础工程化能力，如服务部署、接口调用、模块组织等

### 2.2 当前系统优势

现有项目已经具备较强工程落地性，适合作为实际审核系统原型，具有：

- 可运行
- 可演示
- 可部署
- 可扩展

若用于“原创开源软件赛道”评审，当前版本可能存在以下问题：

1. 学术性不足
2. 创新点不够突出
3. 主要亮点偏工程实现，算法改进不足
4. 对复杂变形文本、混淆文本、规避性输入的处理能力有限

因此，后续迭代目标应聚焦于：

**在不推翻现有系统的前提下，增加研究性功能模块，提高项目的技术深度与创新表达。**

## 3. 迭代目标

本次迭代不重新开发已有功能，而是围绕以下方向对现有系统进行增强：

### 3.1 总体目标

在现有敏感词检测系统基础上，新增若干“增强检测模块”，使系统从“基础敏感词匹配工具”升级为：

**面向变形文本与对抗性输入的增强型敏感词检测系统**

### 3.2 迭代原则

1. 保留现有系统主体架构
2. 尽量复用已有接口、服务层和词库管理模块
3. 新增功能优先采用插件式、可插拔方式接入
4. 优先增加能够体现学术性和创新性的功能
5. 不要求一次性完成全部增强模块，可分阶段迭代

## 4. 本次迭代范围

1. 文本归一化检测增强

## 5. 功能迭代需求

## 5.1 文本归一化增强模块

### 5.1.1 需求背景

当前系统若主要依赖直接匹配，可能难以识别以下变形输入：

- 傻 逼
- 傻-b
- s*b
- s b
- s-b

这类输入本质上仍然表达敏感词，但通过插入符号、空格、分隔符等方式规避检测。

### 5.1.2 迭代目标

在现有检测流程前增加“文本归一化预处理模块”，对输入文本进行统一化清洗，以提高变形词命中率。

### 5.1.3 功能要求

系统应支持以下预处理能力：

1. 去除无意义空格
2. 去除常见分隔符
3. 去除部分无语义干扰字符
4. 统一全角半角字符
5. 英文字母大小写归一化

### 5.1.4 示例

输入：

```text
你这个 s*b 真离谱
```

归一化后：

```text
你这个sb真离谱
```

### 5.1.5 接入方式

建议作为检测流程中的前置模块接入，不影响现有主匹配模块结构。

------

## 5.2 变形敏感词检测增强模块

### 5.2.1 需求背景

仅依赖原始敏感词匹配时，攻击者可以通过插入字符、拆分字符、替换相近字符来规避系统检测。

例如：

- sha bi
- sh@bi
- s4bi
- 傻x
- 傻_
- 傻-逼

### 5.2.2 迭代目标

在现有匹配能力基础上，增加“变形词识别能力”，支持对常见绕过方式进行检测。

### 5.2.3 功能要求

系统应支持以下一种或多种增强策略：

1. 插入字符容忍
2. 特殊符号跳过
3. 常见替代字符映射
4. 拆分写法恢复
5. 相似形式归并后再检测

### 5.2.4 建议实现方式

可采用以下任一方案：

1. 在匹配前完成归一化后再进入现有匹配引擎
2. 对现有 AC 自动机 / DFA / Trie 匹配过程进行扩展
3. 增加二次扫描机制，对初筛未命中内容进行增强判定

### 5.2.5 兼容要求

该模块应尽量不破坏现有系统原有检测逻辑，可作为增强型匹配链路存在。

------

## 5.3 拼音混淆检测模块

### 5.3.1 需求背景

中文敏感词在实际场景中经常被拼音化处理，例如：

- shabi
- sha bi
- ben dan
- caobi

传统中文词典匹配难以识别。

### 5.3.2 迭代目标

在现有系统中新增“拼音混淆检测能力”，识别将中文敏感词转为拼音后的规避文本。

### 5.3.3 功能要求

系统应支持：

1. 敏感词转拼音索引
2. 输入文本中的拼音片段识别
3. 拼音敏感词匹配
4. 中文词与拼音词统一映射到同一风险类别

### 5.3.4 示例

敏感词库中存在：

```text
傻逼
```

则以下文本应尽量被识别：

```text
shabi
sha bi
sha_bi
```

### 5.3.5 建议实现方式

1. 为敏感词构建拼音映射表
2. 增加拼音词表
3. 在归一化后文本中执行拼音匹配
4. 返回命中来源信息，如“原词命中 / 拼音命中”

------

## 5.4 检测结果解释模块

### 5.4.1 需求背景

比赛评审和系统演示时，仅输出“命中/未命中”不够直观。
如果系统能够解释“为什么命中”，更容易体现技术设计与系统价值。

### 5.4.2 迭代目标

增加检测结果解释能力，提高系统可解释性和展示效果。

### 5.4.3 功能要求

检测结果中建议增加以下字段：

1. 命中的敏感词
2. 命中位置
3. 命中方式
4. 是否经过归一化后命中
5. 是否属于拼音混淆命中
6. 风险等级

### 5.4.4 输出示例

```json
{
  "contains_sensitive": true,
  "matched_word": "傻逼",
  "match_type": "pinyin_match",
  "normalized_text": "shabi",
  "risk_level": "high"
}
```

### 5.4.5 价值

该模块有助于：

1. 比赛展示
2. 调试分析
3. 后续论文实验统计
4. 系统可解释性增强

------

## 5.5 检测策略配置模块

### 5.5.1 需求背景

不同场景对检测强度要求不同。
例如：

- 社区评论：要求严格
- 普通聊天：要求适度
- 内部测试：希望关闭部分增强策略

### 5.5.2 迭代目标

让系统支持多种检测策略组合，增强可配置性。

### 5.5.3 功能要求

支持按配置启用/禁用以下模块：

1. 基础精确匹配
2. 文本归一化
3. 变形词检测
4. 拼音检测
5. 高级增强检测

### 5.5.4 示例配置

```json
{
  "exact_match": true,
  "normalize_match": true,
  "pinyin_match": true,
  "fuzzy_match": false
}
```

## 5.6 实验评测支持模块

### 5.6.1 需求背景

如果项目要体现学术性，仅有系统功能还不够，还需要能做实验对比。

### 5.6.2 迭代目标

增加实验与评估支持，用于后续论文、比赛答辩、项目展示。

### 5.6.3 功能要求

支持统计以下指标：

1. 检出率
2. 误报率
3. 漏报率
4. Precision
5. Recall
6. F1 Score
7. 各模块单独贡献效果

### 5.6.4 对比需求

应至少支持以下实验对比：

1. 原始系统 vs 增强系统
2. 无归一化 vs 有归一化
3. 无拼音检测 vs 有拼音检测
4. 仅基础匹配 vs 多策略融合匹配

### 5.6.5 数据集要求

建议准备三类测试样本：

1. 原始敏感文本
2. 变形敏感文本
3. 拼音规避文本
